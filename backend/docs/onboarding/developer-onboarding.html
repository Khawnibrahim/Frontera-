1. Welcome — what this project is (and isn’t)

Welcome to Scalemorph. Your onboarding project is the Frontera Scheduling API — a NestJS backend for healthcare provider scheduling (onboarding, availability, time-off, approvals, calendar exports).

Repo path: /Users/hamzajamshed/scaleMorph/frontera (or your team’s clone of the same repository).

This is a training & delivery project — not production QuoteLogik. Frontera is where you learn our stack, contribute to a real codebase, and grow into cloud-backed backend work. You will not be dropped onto a live customer system on day one.

What Frontera is

  • NestJS 11 + TypeScript on Node 22
  • Supabase Postgres + Drizzle ORM (20+ tables)
  • AWS Serverless — Lambda + API Gateway (serverless.yml)
  • AWS S3 — secure document storage (PACR PDFs, exports)
  • AWS SES — provider invite and notification email
  • Docker Compose — local Postgres + migrate + API with hot reload

Day-one setup

cd scaleMorph/frontera
npm run docker:up          # Postgres + migrations + seed + API on :3000

# Verify
curl http://localhost:3000/health
open http://localhost:3000/api    # Swagger

Alternative: cloud Supabase — see README.md Option B.

Phased delivery (Q1–Q4)

You’ll grow with the roadmap. Today’s screen-backed routes are documented in docs/adr/:

  • Admin: active providers, onboard provider, master availability, schedule change approvals, PRN availability
  • Provider portal: scheduling submissions (PRN + set time-off)
  • Coming: JWT guard (Q1), PACR uploads to S3 (Q2), review queue + SES liaison (Q3), calendar export (Q4)

5. Module 2 — Cloud: file storage (S3) (days 8–12)

Frontera uses AWS S3 for secure documents (PACR PDFs in Q2+) and export artifacts (Q4). Buckets are provisioned in serverless.yml:

  • frontera-{stage}-secure-documents → DOCUMENTS_BUCKET
  • frontera-{stage}-exports → EXPORTS_BUCKET

5.1 Gateway pattern

S3Gateway wraps the AWS SDK v3 client. Domain services depend on IAwsS3Gateway, not the SDK directly — same pattern you’ll use for any future OCR result storage.

// Mental model
provider-documents.service
  → S3Gateway.putObject({ bucket, key, body, contentType })
  → documents row in Postgres (metadata + S3 key)

5.2 Concepts to master

  • Bucket vs key vs region
  • IAM permissions on Lambda role (PutObject, GetObject)
  • Pre-signed URLs (future download flows)
  • Never expose bucket credentials to the browser — server uploads only

5.3 Exercises

  • Read s3.gateway.ts and s3.interface.ts — list what’s missing for GetObject
  • Trace provider-documents.service.ts (if present) or read Q2 notes in CONTEXT.md
  • Find bucket names in serverless.yml and match to .env.example
  • Write 3 failure modes for an upload (network, IAM, wrong bucket) and expected behaviour

6. Module 3 — Cloud: compute (Docker + Serverless) (days 13–17)

6.1 Docker — local development

postgres:16  →  migrate (db:migrate + seed)  →  api (nest start --watch)
                      ↑                                    |
                      └──── DATABASE_URL (no SSL) ─────────┘

See docs/docker.md. Only ./src is bind-mounted — rebuild image after package.json changes.

6.2 Serverless — production

  • Service: frontera-scheduling
  • Function: api → dist/src/lambda.handler
  • Runtime: Node 22, timeout 300s, HTTP API {proxy+}
  • Deploy: npm run deploy (build + serverless deploy)

6.3 Docker vs Serverless — when to use which

Docker (local / optional host)Lambda (deployed API)
Fast iteration, hot reloadProduction, scales with API Gateway
Full control over processPay per request; cold starts possible
Postgres in containerConnects to Supabase pooler in cloud
Good for learning & integration testsSame Nest code via lambda.ts
Future async work: bulk calendar export or heavy email batches should become enqueue → 202 Accepted → consumer Lambda (SQS). Don’t block the API Lambda on long jobs. SQS_QUEUE_URL is reserved in .env.example but not wired yet.

6.4 Exercises

  • Run npm run docker:reset once — understand migrate + seed
  • Open src/lambda.ts and src/main.ts — find shared createApp()
  • Skim serverless.yml — list every AWS resource it creates
  • Explain in writing: what breaks if FRONTERA_AWS_REGION doesn’t match deploy region

7. Module 4 — Messaging, documents & OCR concepts (days 15–19)

7.1 SES (email)

Provider invites and liaison notifications go through SesGateway. Deploy creates SES identity for SES_FROM_EMAIL. Read docs/onboarding-invite-flow.md.

Invite flow (simplified)
  Admin creates invite
    → Nest renders HTML email (templates/)
    → SES sends with link to FRONTERA_API_PUBLIC_URL/accept-invite
    → Provider sets password via Supabase Auth

7.2 PACR documents (Q2)

Post-deadline schedule changes require a PACR PDF. Backend must validate pacr_document_id, store file in S3, link documents table. This is your document pipeline exercise — upload, metadata, authorization.

7.3 OCR — learning path (not in Frontera yet)

Frontera does not call Azure/Textract today. As a Scalemorph developer you should still understand cloud OCR for other projects:

ConceptTypical patternRelates to Frontera when…
Submit documentPOST bytes to cloud OCR APIAuto-extract PACR fields (future)
Async poll202 + Operation-Location → poll until doneLarge PDFs can’t block HTTP
Store textPersist in DB or object metadataSearchable document audit trail
Sync vs queueInline in request vs SQS workerSame trade-off as calendar export

Stretch exercise (offline): read Azure Document Intelligence or AWS Textract “analyze document” docs and write a 1-page comparison — no code required in Frontera repo.

8. Module 5 — Security (JWT, RLS, secrets) (days 18–22)

8.1 Authentication (Q1 — read now, implement soon)

Supabase Auth issues JWT → Lovable sends Authorization: Bearer <token> → Nest guard verifies via SUPABASE_JWT_SECRET or JWKS. Full detail: docs/api-auth.md.

Routes are open in local dev until the guard ships. Always design services assuming userId and role will be on the request — don’t bake in “no auth forever”.

8.2 Row-level security (Postgres)

RLS helpers in drizzle/0001_supabase_rls_functions.sql: has_role, get_user_org_ids, is_assigned_to, log_audit. Nest uses pooler connection — repository queries must still respect org/site scope when guard is live.

8.3 Secrets — never commit

VariableRisk if leaked
DATABASE_URLFull database access
SUPABASE_JWT_SECRETForged tokens
SUPABASE_SERVICE_ROLE_KEYBypass all RLS
AWS_ACCESS_KEY_ID / profileS3/SES abuse, deploy rights

10. Module 7 — Critical reasoning (don’t mind-dump on AI) (daily, all 30 days)

AI makes you fast; it does not make you correct. On Frontera, wrong scheduling rules or leaked PACR paths have real compliance implications — you must understand what you ship.

10.1 Daily habits (15–30 min)

HabitAction
Read before ask3-sentence summary of a file before Claude explains it
Predict before runWrite expected API response; compare to Swagger/curl
Explain aloudWeekly: explain invite flow or deadline rule without notes
Bug journalExpected / actual / hypothesis / fix — you write it

10.2 Frameworks

5 Whys (example)

  1. Why was time-off rejected? → Missing PACR
  2. Why required? → Past scheduling deadline
  3. Who enforces deadline? → Nest service (not frontend alone)
  4. What date logic? → Last Tuesday of M−2
  5. Where is helper? → Find it in codebase before changing

Teach-back gate (before every PR)

  • What user role can call this endpoint?
  • Which tables change?
  • What happens on validation failure?
  • How did you test without AI writing the test for you?
